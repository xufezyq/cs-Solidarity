大家好，这次介绍的是 cs-Solidarity。
一个基于 Windows 微信 PC 端的多功能自动化机器人。<#0.50#>

它的底层不是 hook 微信进程，也不是直接读写微信内存。
而是基于 wxauto、Win32 GUI 自动化和窗口控制能力。
像真实用户一样操作微信窗口。<#0.50#>

从功能上看，它能做 Steam 监控、AI 对话、AI Agent 自动化、信息推送、灾害预警和 Web 远程管理。<#0.40#>

从工程上看，它更像一个微信自动化框架。
不同功能模块可以独立运行。
但所有真正操作微信的动作，都会被收束到统一的主线程和反检测机制里。<#1.00#>

理解这个项目，可以先抓住三个关键词。<#0.40#>

第一个，是线程安全。
微信 GUI 自动化，不能让多个后台线程同时操作窗口。
所以项目把所有实例要发送的消息先放进队列。
再由主线程统一发送。<#0.50#>

第二个，是反检测。
自动化如果每次点击都太直、太准、太规律，就很容易暴露机器行为。
项目在鼠标、键盘、窗口切换和等待时间上，都加入了更接近真人操作的随机性。<#0.50#>

第三个，是远程管理。
Bot 可以跑在内网 Windows 电脑上，Web Server 跑在公网 VPS 上。
Agent 主动向外连接 Server。
这样浏览器就能远程管理 Bot，同时不需要给内网机器做端口映射。<#0.60#>

后面所有功能，其实都围绕这三个设计展开。<#1.00#>

项目目前可以概括成四大核心能力。<#0.40#>

第一，游戏状态监控。
它会定时拉取 Steam 好友状态。
判断谁开始游戏、谁结束游戏，并把变化推送到微信群。<#0.40#>

第二，智能对话和 Agent 编排。
它支持 OpenAI 兼容接口。
既可以用命令前缀触发 ChatAuto，把自然语言任务交给 AI Agent。
也可以像 KouriChat 一样监听群聊上下文，做更自然的长期对话。<#0.50#>

第三，信息与预警推送。
除了日常资讯、金价、股票，还可以接入 CS 新闻日报、地震、气象和海啸预警。<#0.40#>

第四，远程管理面板。
它提供 Dashboard、日志、配置、文件、实例和控制能力。
让这个 Bot 不只是能跑，还能被远程维护。<#1.00#>

系统全景可以拆成三部分。<#0.40#>

第一部分是外部数据源和外部工具。
比如 Steam Web API、完美世界、AI 模型接口、AI Agent、财经接口、灾害预警数据源。
以及像 CS2-insight-agent 这样的 demo 解析和视频处理工具。<#0.50#>

第二部分是 Bot 主程序。
它负责实例调度、消息队列、消息路由、本地 Bot API，以及最终的微信窗口操作。<#0.40#>

第三部分是 Web 管理部分。
Agent 跑在内网 Windows 上，Server 跑在公网 VPS 上。
浏览器访问 Server，再由 Server 通过 WebSocket，把操作转发给 Agent。<#0.50#>

最关键的一点是：
不管消息来自 Steam、AI、Agent、灾害预警，还是 Web 面板。
最后真正操作微信的动作，都必须回到 Bot 主线程。<#0.60#>

这就是为什么这些能力，可以共用一个微信自动化核心。<#1.00#>

项目内部可以拆成三层。<#0.40#>

第一层是 Core 框架层。
这里包括 main.py 主循环、BaseInstance 基类、实例工厂、消息队列和反检测 Patch。
这一层决定整个系统怎么启动、怎么调度、怎么收发消息。<#0.50#>

第二层是 Instances 功能层。
SteamAuto、ChatAuto、KouriChat、InfoPush、DailyAuto、DisasterWarning，都是独立实例。
它们各自处理业务逻辑，但不直接抢占微信窗口。<#0.50#>

第三层是 External 接入层。
它负责连接 Steam、完美世界、AI 接口、AI Agent、Bot 本地 API、财经数据、灾害预警和 Web Panel。<#0.50#>

这个分层的好处是，新功能不需要硬塞进主循环。
你只需要写一个新的 BaseInstance 子类，注册到工厂。
再在配置里添加实例，就能接入整个系统。<#1.00#>

项目目前包括六种实例。<#0.40#>

SteamAuto 负责游戏状态、CS2 新闻、完美战绩和每日榜单。<#0.30#>

ChatAuto 不只是负责命令式 AI 对话。
它更重要的作用，是把微信群里的自然语言请求，变成 AI Agent 任务。
先通过大语言模型接口理解用户意图。
再让 Agent 调用 Bot API，把结果发回对应群聊。<#0.50#>

这让微信不只是一个聊天入口，而是变成了 Agent 的交互界面。
它可以承接定时汇总、文件处理、内容生成和结果回传这类工作流。
再通过 Bot API，把处理结果发回对应群聊。<#0.60#>

KouriChat 则是另一类 AI 能力。
它不是命令入口，而是更接近“常驻群友”的自然聊天机器人。
它可以按群配置不同人设，读取群聊上下文。
结合记忆系统和图片识别来判断是否回复。
有的群只在艾特或触发词命中时回复。
有的群可以配置成更主动的陪伴式对话。<#0.50#>

这也能看出 ChatAuto 和 KouriChat 的区别：
ChatAuto 负责把明确命令交给 Agent。
KouriChat 负责更自然地参与聊天。<#0.50#>

InfoPush 负责金价、股票、每日新闻这类定时信息。<#0.30#>

DailyAuto 负责固定时间发送固定消息，比如每日提醒。<#0.30#>

DisasterWarning 负责灾害预警，把不同来源的数据整理后推送到对应会话。<#0.40#>

这些实例看起来功能差异很大。
但它们接入主程序的方式是一致的。
后台处理业务，发送时进入队列。
或者由 Agent 调用 Bot API 入队。
最后都由主线程操作微信。<#1.00#>

SteamAuto 是最容易理解的一个实例。<#0.40#>

它会定时调用 Steam Web API，拿到好友当前状态。
然后把这一轮状态和上一轮缓存做对比。<#0.40#>

如果某个好友的游戏状态从空变成 730，就说明他开始玩 CS2。
如果从 730 变为空，就说明他离开了 CS2。
机器人就可以生成一条群聊通知。<#0.50#>

这些进入游戏、离开游戏和战绩播报消息，不是人工写死的文案。
而是代码根据 Steam 状态变化、游戏名称和计时结果生成的。<#0.50#>

对于 CS2，项目还可以继续接入完美世界平台。
查询最近比赛数据。
并把结果整理成适合群聊阅读的播报。<#0.40#>

每天的定时任务，还会把这些数据合成一张标准日报图片。
它不是手工排版，而是代码模板自动生成。
里面包含好友今日游玩统计、完美平台统计，以及完整的历史战绩排行榜。<#0.50#>

关键点是：
SteamAuto 自己不直接操作微信。
它只负责发现状态变化，然后调用发送消息函数。
这个函数已经被主程序替换成“放入发送队列”。
所以真正发微信的动作，仍然回到主线程执行。<#1.00#>

微信 GUI 自动化最容易出问题的地方，是行为太像机器。<#0.40#>

比如鼠标每次都走直线，点击位置完全一样。
键盘输入间隔固定，窗口一直保持前台。
这些都不是正常用户的操作习惯。<#0.50#>

所以项目在底层做了一组反检测 Patch。<#0.40#>

鼠标移动使用贝塞尔曲线路径，并加入高斯扰动。
点击前后会有随机等待。
键盘输入会模拟按键之间的时间差。
窗口显示会移除不自然的置顶状态。
发送完成后，还会切回安全聊天并最小化微信。<#0.50#>

更重要的是，这些逻辑不需要每个实例或者 Agent 自己实现。
SteamAuto、ChatAuto、KouriChat 只关心业务。
外部 Agent 只需要调用 Bot API。
真正“像人一样操作微信”的细节，由 Core 层统一处理。<#1.00#>

主循环可以理解成五步。<#0.40#>

第一步，检查维护窗口。
默认在维护时间段减少主动操作，降低异常和无效动作。<#0.35#>

第二步，处理 Web 消息。
来自远程面板的指令会先进入队列，再走统一逻辑。<#0.35#>

第三步，消费发送队列。
后台实例产生的所有待发送消息，都会在这里由主线程统一发送。<#0.35#>

第四步，接收和分发新消息。
程序检测微信新消息，再根据路由规则交给不同实例处理。<#0.35#>

第五步，空闲时最小化微信窗口。<#0.50#>

主循环的顺序很重要。
项目优先处理待发送消息，再去读取新消息。
这样可以减少切换聊天窗口时，未读状态被清掉的竞态问题。<#1.00#>

消息进入系统后，会分成两种路由方式。<#0.40#>

第一种是独占实例。
比如 ChatAuto 只处理带固定前缀的消息。
只要前缀匹配，这条消息就交给它，不再广播给其他实例。
后续它可以直接调用大语言模型回复。
也可以把任务交给 AI Agent，再由 Agent 通过 Bot API 发回结果。<#0.50#>

如果用户发出明确命令。
这条消息就应该直接交给 ChatAuto。
如果用户只是普通聊天，消息才会继续广播给 KouriChat 这类上下文监听模块。<#0.50#>

第二种是广播实例。
普通群聊消息会分发给需要全量监听的模块，比如 KouriChat。
这样它可以观察上下文，再决定是否参与对话。<#0.50#>

这个设计解决的是：
多个功能同时监听一个群聊时，谁应该处理消息的问题。
命令式功能和自然对话式功能可以共存，而且不会互相抢消息。<#1.00#>

远程管理这部分，核心是 Agent 主动连出。<#0.40#>

Bot 所在机器通常在内网，没有公网 IP。
如果要求用户做端口映射或者内网穿透，部署会很麻烦，也会增加安全风险。<#0.50#>

这个项目的做法是：
Agent 跑在内网 Windows 上，主动通过 WebSocket 连接公网 Server。
浏览器只需要访问 Server。
Server 再把请求转发给 Agent。<#0.50#>

所以在 Web 面板里，我们可以看到 Bot 状态、运行时间、实例状态和实时日志。
也可以修改配置、上传文件、执行启动、停止和重启。<#0.50#>

这说明 Bot 不是只能在本机控制。
而是可以远程观测、远程配置、远程运维。<#1.00#>

Web 面板把运维能力集中在一起。<#0.40#>

Dashboard 用来看 Agent 和 Bot 是否在线。
Config 用来编辑 JSON 配置，并支持备份恢复。
Control 用来执行启动、停止和重启。
Logs 用来查看实时日志和历史日志。
Files 用来管理 Web 端或 Agent 端上传文件。
Chat 用来把网页消息转发给 Bot 实例，并可选择同步到微信，或仅在网页显示。<#0.60#>

如果从网页 Chat 发出一条消息。
后面会依次经过 Web 请求、Agent 转发、Bot 发送队列。
最后由微信窗口完成发送。
这样网页控制不是一个独立面板，而是真的接到了 Bot 的发送通道上。<#0.60#>

对一个长期运行的机器人来说，功能本身很重要。
但可观测和可维护同样重要。
否则一旦 Bot 跑在另一台电脑上，出问题就只能远程桌面，或者手动查日志。
维护成本会很高。<#1.00#>

总结一下。<#0.40#>

cs-Solidarity 表面上是一个微信机器人。
但它真正的工程价值，是把微信 GUI 自动化做成了一套框架。<#0.50#>

第一，线程安全。
后台实例只负责业务逻辑和消息入队。
真正操作微信的动作，统一回到主线程。<#0.40#>

第二，模块化扩展。
Steam、AI、信息推送、灾害预警，都可以作为独立实例接入。
不需要破坏主循环。<#0.40#>

第三，远程可运维。
Agent 主动连接 Server。
让内网 Windows 上的 Bot，也可以通过公网 Web 面板管理。<#0.50#>

如果你关注微信自动化、游戏数据推送、AI 群聊机器人。
或者想看一个 GUI 自动化项目，怎么做成可维护的系统。
这个项目会是一个很好的参考。<#0.80#>
